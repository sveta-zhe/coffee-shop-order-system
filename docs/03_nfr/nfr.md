# Нефункциональные требования

## 1. Производительность (Performance)

### 1.1. Нагрузка

**Данные для расчета:**
- Сеть кофеен: 5 точек
- Среднее количество заказов в день на точку: 200
- Часы пик: 8:00–10:00 и 12:00–14:00 (4 часа)

**Расчет RPS (запросов в секунду):**

| Показатель | Формула | Значение |
| :--- | :--- | :--- |
| Заказов в сутки (всего) | 5 точек × 200 заказов | 1000 заказов |
| Заказов в пик | 80% от 1000 | 800 заказов |
| Длительность пика | 4 часа × 3600 секунд | 14 400 секунд |
| **RPS_peak** | 800 / 14 400 | **~0.056 запросов/сек** |

> **Вывод:** Нагрузка очень низкая. Один сервер справится с запасом.

### 1.2. Latency (время ответа)

| Сценарий | Целевое p95 | Метод измерения |
| :--- | :--- | :--- |
| Создание заказа | < 500 мс | Замер на API gateway |
| Получение заказа по ID | < 200 мс | Замер на API gateway |
| Оплата | < 2 секунд | Включая время ответа платежного шлюза |

### 1.3. Concurrency (одновременные запросы)

Формула: `Concurrency = RPS_peak × AvgLatency`

- RPS_peak = 0.056
- AvgLatency (создание заказа) = 0.3 сек

**Concurrency ≈ 0.056 × 0.3 ≈ 0.017**

> **Вывод:** Система практически не испытывает конкуренции. Очередей не будет.

## 2. Доступность (Availability)

| Уровень | Целевое значение | Допустимый простой в год | Допустимый простой в месяц |
| :--- | :--- | :--- | :--- |
| **SLO** | 99.9% (три девятки) | 8.76 часов | 43.8 минут |

**Определение "down":** Система считается недоступной, если более 5% запросов за 5-минутное окно возвращают ошибку 5xx.

**Что входит в расчет:**
- Основной API (создание заказа, получение статуса)
- Платежный шлюз (исключая его собственные сбои)

**Что НЕ входит:**
- Запланированные технические работы (уведомление за 24 часа)

---

## 3. Надежность (Reliability)

| Показатель | Целевое значение | Как измерять |
| :--- | :--- | :--- |
| **MTBF** (среднее время между сбоями) | > 720 часов (30 дней) | По логам ошибок |
| **Частота отказов** | < 0.1% запросов | По метрикам API |

**Допустимые сбои:**
- Кратковременная недоступность платежного шлюза — система должна повторить запрос
- Внутренняя ошибка сервера — не более 1 раза в неделю

---

## 4. Поддерживаемость (Maintainability)

| Показатель | Целевое значение |
| :--- | :--- |
| **MTTR** (среднее время восстановления) | < 30 минут |
| **Время развертывания новой версии** | < 10 минут |

**Практики снижения MTTR:**
- Автоматическое логирование ошибок
- Дашборд с ключевыми метриками (Grafana)
- Оповещения в Telegram при падении API

---

## 5. Масштабируемость (Scalability)

**Текущая нагрузка:** 1000 заказов/день  
**Целевая нагрузка через год:** 2000 заказов/день  

**Стратегия масштабирования:**
- Вертикальное (увеличить мощность сервера) — до 5000 заказов/день
- При превышении — горизонтальное (добавить второй сервер + балансировщик)

**Индикатор для масштабирования:** CPU сервера > 70% в течение 30 минут

---

## 6. Безопасность (Security)

| Требование | Реализация |
| :--- | :--- |
| Аутентификация | JWT-токены, срок жизни 24 часа |
| Авторизация | Ролевая модель (клиент, бариста, управляющий) |
| Шифрование | HTTPS для всех API-запросов |
| Хранение данных | Пароли только в хешированном виде (bcrypt) |
| Логирование | Доступ к логам только у администратора |

---
