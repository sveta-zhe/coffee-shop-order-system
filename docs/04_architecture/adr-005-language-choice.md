# ADR-005: Выбор языка программирования

## Статус
Принято

## Контекст
Команда: 1–2 разработчика (на старте).  
Система: небольшое монолитное приложение с REST API и базой данных.  
Нагрузка: низкая (до 2000 заказов/день).

## Варианты решения

### Вариант 1. Python (FastAPI / Django)
**Описание:** Высокоуровневый язык с динамической типизацией, огромной экосистемой.

**Плюсы:**
- Быстрая разработка MVP (в 2–3 раза быстрее Java)
- Огромное количество библиотек (SQLAlchemy, Pydantic, Alembic)
- Легко найти разработчиков
- Идеален для прототипирования

**Минусы:**
- Медленнее Java и Go (но для нашей нагрузки — не критично)
- Динамическая типизация может привести к ошибкам в рантайме

### Вариант 2. Java (Spring Boot)
**Описание:** Статически типизированный, компилируемый язык, стандарт в корпоративной разработке.

**Плюсы:**
- Высокая производительность
- Строгая типизация (меньше ошибок)
- Мощные фреймворки (Spring)
- Подходит для крупных систем

**Минусы:**
- Медленная разработка (больше кода)
- Сложнее для новичков
- Больше "магии" в фреймворках

### Вариант 3. Go
**Описание:** Компилируемый язык с простым синтаксисом, создан для высоконагруженных систем.

**Плюсы:**
- Очень высокая производительность
- Простой синтаксис
- Отличная поддержка конкурентности

**Минусы:**
- Мало библиотек для бизнес-логики (по сравнению с Python/Java)
- Сложнее писать прототипы
- Меньше разработчиков на рынке

## Решение
Выбран **Python с FastAPI**.

**Обоснование:**
1. Команда знает Python (я учу его и понимаю)
2. Нагрузка низкая — производительность не критична
3. Нужно быстро сделать MVP — Python идеален
4. FastAPI дает автодокументацию Swagger (бесплатно)
5. SQLAlchemy — лучший ORM для работы с PostgreSQL

## Последствия
- Использовать типизацию (mypy) для контроля ошибок
- Использовать Pydantic для валидации данных
- При росте нагрузки можно переписать горячие участки на Go (но это маловероятно)
