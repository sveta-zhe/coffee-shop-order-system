# ADR-004: Выбор базы данных

## Статус
Принято

## Контекст
Система управляет заказами, продуктами, платежами. Данные имеют четкие связи:
- У одного заказа много позиций
- У одной позиции один продукт
- У одного заказа один платеж

Ожидаемая нагрузка: низкая (до 2000 заказов/день).

## Варианты решения

### Вариант 1. PostgreSQL (реляционная БД)
**Описание:** Классическая реляционная база данных с поддержкой SQL, транзакций, внешних ключей.

**Плюсы:**
- Поддержка сложных связей между таблицами (foreign keys)
- Транзакции ACID (нельзя потерять платеж)
- SQL — стандартный язык, знают все разработчики
- Отличная поддержка для Python (psycopg2, SQLAlchemy)

**Минусы:**
- Сложнее масштабировать горизонтально (но нам не нужно)
- Схема данных жесткая (менять сложнее)

### Вариант 2. MongoDB (NoSQL, документо-ориентированная)
**Описание:** Хранит данные в виде JSON-документов, не требует схемы.

**Плюсы:**
- Гибкая схема (можно менять на лету)
- Легко масштабируется горизонтально
- Быстрый доступ к вложенным данным

**Минусы:**
- Нет транзакций (риск потерять данные при сбое)
- Нет внешних ключей (связи нужно контролировать в коде)
- Сложные отчеты требуют агрегации

## Решение
Выбран **PostgreSQL**.

**Обоснование:**
1. У нас четкие связи между сущностями — реляционная модель идеально подходит
2. Нужны транзакции (оплата и создание заказа должны быть атомарны)
3. Нужны сложные отчеты (выручка по часам, топ продуктов) — SQL справляется лучше
4. Нагрузка низкая — масштабирование не требуется
5. Python + PostgreSQL — стандартный стек, много документации

## Последствия
- Проектировать схему БД заранее (но это плюс)
- При росте нагрузки можно добавить реплики для чтения
- Индексы на часто запрашиваемые поля (created_at, status)
