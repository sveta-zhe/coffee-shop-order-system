# ADR-001: Выбор архитектурного стиля

## Статус
Принято

## Контекст
Необходимо спроектировать систему управления заказами для сети кофеен.  
Ожидаемая нагрузка: до 2000 заказов в день (≈0.1 RPS в пик).  
Команда: 1–2 разработчика (на старте).  
Бюджет: ограниченный.  
Сроки: MVP за 3 месяца.

## Варианты решения

### Вариант 1. Монолит
**Описание:** Одно приложение, включающее все компоненты (API, бизнес-логика, работа с БД).

**Плюсы:**
- Простота разработки и развертывания
- Легко тестировать (одна точка входа)
- Меньше накладных расходов на сетевое взаимодействие
- Подходит для небольшой команды

**Минусы:**
- Сложно масштабировать отдельные компоненты
- При росте системы может стать «большим комом»
- Ошибка в одном модуле может уронить всё

### Вариант 2. Микросервисы
**Описание:** Разделение системы на независимые сервисы (заказы, оплата, уведомления, отчеты).

**Плюсы:**
- Масштабирование отдельных сервисов
- Независимость технологий
- Изоляция ошибок

**Минусы:**
- Сложность разработки (сетевое взаимодействие, консистентность)
- Требуется DevOps для развертывания
- Накладные расходы на межсервисные вызовы

## Решение
Выбран **модульный монолит**.

**Обоснование:**
1. Нагрузка крайне низкая (0.1 RPS) — масштабирование не требуется
2. Команда маленькая — монолит проще разрабатывать и поддерживать
3. Бюджет ограничен — микросервисы потребуют дополнительных ресурсов
4. Модульный монолит позволяет в будущем вынести компоненты в отдельные сервисы без полной переписывания

## Последствия
- В коде нужно четко выделять модули (заказы, оплата, уведомления)
- Использовать четкие интерфейсы между модулями
- При росте нагрузки можно будет вынести модули в отдельные сервисы
- Единая база данных (пока), но с четким разделением таблиц по модулям
