# ADR-003: Стратегия кэширования

## Статус
Принято

## Контекст
API часто возвращает список продуктов (напитки, добавки). Эти данные меняются редко (раз в день).  
Каждый клиент, открывая страницу заказа, запрашивает этот список.

## Варианты решения

### Вариант 1. Без кэша (каждый раз из БД)
**Описание:** Каждый запрос списка продуктов читает из базы данных.

**Плюсы:**
- Всегда актуальные данные
- Простота реализации

**Минусы:**
- Лишняя нагрузка на БД
- При высоком RPS может стать узким местом

### Вариант 2. Cache-Aside
**Описание:** Приложение сначала проверяет кэш (Redis). Если данных нет — читает из БД и кладет в кэш.

**Плюсы:**
- Снижает нагрузку на БД
- Ускоряет ответы (до 10–20 раз)
- Простая логика инвалидации

**Минусы:**
- Нужен дополнительный сервис (Redis)
- Данные могут быть немного устаревшими

### Вариант 3. Read-Through
**Описание:** Кэш сам загружает данные из БД при промахе.

**Плюсы:**
- Приложение не знает о БД
- Единая точка доступа

**Минусы:**
- Сложнее реализовать
- Некоторые кэши не поддерживают этот паттерн

## Решение
Выбран **Cache-Aside** с Redis.

**Обоснование:**
1. Список продуктов меняется редко — кэш будет эффективен
2. Redis легковесен и прост в настройке
3. Инвалидация простая: при обновлении продукта удаляем ключ из кэша
4. Нет требований к абсолютной свежести данных (допустима задержка в 1–2 секунды)

## Последствия
- Добавить Redis в архитектуру
- Время жизни кэша (TTL) установить: 1 час
- При обновлении продукта инвалидировать кэш (удалить ключ)
- При старте системы прогреть кэш (загрузить список продуктов)
