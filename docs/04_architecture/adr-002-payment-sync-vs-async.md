# ADR-002: Обработка платежей

## Статус
Принято

## Контекст
При создании заказа клиент должен оплатить его. Платежный шлюз может отвечать медленно (до 2 секунд).  
В часы пик количество заказов может достигать 800 за 4 часа (≈0.056 RPS).

## Варианты решения

### Вариант 1. Синхронная оплата (REST)
**Описание:** API вызывает платежный шлюз и ждет ответа. Клиент получает результат сразу.

**Плюсы:**
- Простота реализации
- Клиент сразу знает результат оплаты
- Нет необходимости в дополнительных компонентах

**Минусы:**
- Если шлюз тормозит, клиент ждет (плохой UX)
- При ошибке шлюза запрос может зависнуть

### Вариант 2. Асинхронная оплата (очередь)
**Описание:** API сохраняет запрос на оплату в очередь и сразу возвращает клиенту «в обработке». Отдельный воркер обрабатывает платежи.

**Плюсы:**
- Клиент не ждет — интерфейс отзывчивый
- Легче переживает сбои платежного шлюза
- Можно повторять неудачные платежи

**Минусы:**
- Сложнее реализовать
- Нужна очередь (RabbitMQ/Kafka)
- Клиент получает статус с задержкой

## Решение
Выбрана **синхронная оплата** с таймаутом и ретраями.

**Обоснование:**
1. Нагрузка низкая — очередь не даст выигрыша
2. Клиенту важно сразу знать, оплачен ли заказ (чтобы не стоять в очереди дважды)
3. Синхронный подход проще для MVP
4. В будущем можно перейти на асинхронную, если платежный шлюз начнет тормозить

## Последствия
- Установить таймаут на вызов платежного шлюза: 5 секунд
- При ошибке — повторить запрос 3 раза с экспоненциальной задержкой
- При финальной ошибке — вернуть клиенту сообщение «Оплата не прошла, попробуйте позже или оплатите на месте»
